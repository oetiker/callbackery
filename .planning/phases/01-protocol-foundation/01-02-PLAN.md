Create surface schemas following A2UI pattern.

Surfaces are renderable containers where UI appears. Each surface has its own component tree and data model scope.

Schema definitions:

1. **SurfaceId** - String identifier for surfaces:
   ```yaml
   type: string
   description: Unique identifier for a surface
   examples: ["main", "modal-1", "toast-area"]
   ```

2. **SurfaceType** - Types of surfaces (string, open set):
   ```yaml
   type: string
   description: Surface presentation type (open set)
   examples: ["main", "modal", "sidebar", "toast", "panel"]
   ```

3. **Surface** - Complete surface definition:
   ```yaml
   type: object
   required: [id, type]
   properties:
     id:
       $ref: '#/components/schemas/SurfaceId'
     type:
       $ref: '#/components/schemas/SurfaceType'
     title:
       type: string
       description: Surface title (for modals, panels)
     componentTree:
       $ref: 'component.yaml#/components/schemas/ComponentTree'
       description: UI structure for this surface
     data:
       $ref: 'data.yaml#/components/schemas/DataModel'
       description: Data model scoped to this surface
   ```

4. **CreateSurfaceRequest** - Request to create a surface:
   ```yaml
   type: object
   required: [id, type]
   properties:
     id:
       $ref: '#/components/schemas/SurfaceId'
     type:
       $ref: '#/components/schemas/SurfaceType'
     title:
       type: string
     componentTree:
       $ref: 'component.yaml#/components/schemas/ComponentTree'
     data:
       $ref: 'data.yaml#/components/schemas/DataModel'
   ```

5. **UpdateSurfaceRequest** - Request to update surface:
   ```yaml
   type: object
   properties:
     title:
       type: string
     componentTree:
       $ref: 'component.yaml#/components/schemas/ComponentTree'
     data:
       $ref: 'data.yaml#/components/schemas/DataModel'
   description: Only provided fields are updated
   ```

Include description explaining:
- Surfaces separate "where to show" from "what to show"
- Each surface has isolated component tree and data scope
- Main surface is always present, others (modal, toast) are created/destroyed
  </action>

- File exists at spec/openapi/schemas/surface.yaml
- Contains Surface, SurfaceId, SurfaceType schemas
- Surface has componentTree and data references to Plan 01 schemas
  </verify>

Create action (user to backend) and event (backend to frontend) schemas.

**Action Schema** (action.yaml):

Actions represent user intent. Frontend captures, backend decides response.

1. **ActionType** - Type of user action (open set):
   ```yaml
   type: string
   description: Action type identifier (open set)
   examples: ["submit", "click", "navigate", "select", "dismiss"]
   ```

2. **Action** - User action message:
   ```yaml
   type: object
   required: [type, source]
   properties:
     type:
       $ref: '#/components/schemas/ActionType'
       description: What kind of action
     source:
       $ref: 'component.yaml#/components/schemas/ComponentId'
       description: Component that triggered the action
     surfaceId:
       $ref: 'surface.yaml#/components/schemas/SurfaceId'
       description: Surface containing the source component
     context:
       type: object
       additionalProperties: true
       description: |
         Resolved data context at action time.
         Backend defines what data to include via component configuration.
         Frontend resolves JSON Pointer paths to actual values.
     timestamp:
       type: string
       format: date-time
   ```

3. **ActionRequest** - API request containing action:
   ```yaml
   type: object
   required: [action]
   properties:
     action:
       $ref: '#/components/schemas/Action'
   ```

4. **ActionResponse** - Backend response to action:
   ```yaml
   type: object
   required: [success]
   properties:
     success:
       type: boolean
     updateSurface:
       $ref: 'surface.yaml#/components/schemas/UpdateSurfaceRequest'
       description: Surface updates to apply
     createSurface:
       $ref: 'surface.yaml#/components/schemas/CreateSurfaceRequest'
       description: New surface to create (e.g., modal)
     closeSurface:
       $ref: 'surface.yaml#/components/schemas/SurfaceId'
       description: Surface to close
     navigate:
       type: string
       description: URL or route to navigate to
     error:
       $ref: 'error.yaml#/components/schemas/ApiError'
   ```

**Event Schema** (event.yaml):

Events are backend signals. They are triggers, not data carriers.

1. **EventType** - Type of event (open set):
   ```yaml
   type: string
   description: Event type identifier (open set)
   examples: ["data_changed", "config_changed", "session_expired", "notification"]
   ```

2. **Event** - Backend event message:
   ```yaml
   type: object
   required: [type]
   properties:
     type:
       $ref: '#/components/schemas/EventType'
       description: What kind of event
     surfaceId:
       $ref: 'surface.yaml#/components/schemas/SurfaceId'
       description: Surface this event relates to (if applicable)
     payload:
       type: object
       additionalProperties: true
       description: |
         Event-specific data. Keep minimal - events are triggers.
         Frontend should re-fetch data via REST, not rely on event payload.
     timestamp:
       type: string
       format: date-time
   ```

3. **Notification** - User-facing notification event:
   ```yaml
   type: object
   required: [level, message]
   properties:
     level:
       type: string
       enum: [info, success, warning, error]
     message:
       type: string
     title:
       type: string
     duration:
       type: integer
       description: Auto-dismiss after milliseconds (0 = manual dismiss)
   ```

Include descriptions explaining:
- Actions power everything: navigation, forms, tracking, operations
- Events are triggers - frontend re-fetches via REST
- Context in actions is resolved at send time (not raw paths)
  </action>
  <verify>
- Files exist at spec/openapi/schemas/action.yaml and event.yaml
- Action has type (open string), source, context
- Event has type (open string), payload
- ActionResponse can trigger surface updates, navigation
  </verify>
  <done>Action and event message schemas defined with open type sets</done>

  <name>Task 4: Define schema discovery endpoint</name>
  <files>spec/openapi/paths/schemas.yaml</files>
  <action>
Create schema discovery endpoint for frontend-backend schema exchange.

Pattern from research: Frontend publishes component schemas, backend fetches what it needs to validate.

```yaml
/schemas:
  get:
    summary: List available schemas
    operationId: listSchemas
    description: |
      Returns list of available schema categories.
      Frontend publishes schemas, backend can fetch to validate messages.
    responses:
      '200':
        description: Available schema categories
        content:
          application/json:
            schema:
              type: object
              properties:
                categories:
                  type: array
                  items:
                    type: string
                  examples:
                    - ["components", "actions", "events"]

/schemas/components:
  get:
    summary: Get component type schemas
    operationId: getComponentSchemas
    description: |
      Frontend publishes JSON Schemas for each component type it supports.
      Backend validates component configurations against these schemas.
    responses:
      '200':
        description: Map of component type to JSON Schema
        content:
          application/json:
            schema:
              type: object
              additionalProperties:
                type: object
                description: JSON Schema for component type
              example:
                text-field:
                  type: object
                  properties:
                    label: { type: string }
                    placeholder: { type: string }
                    required: { type: boolean }
                button:
                  type: object
                  properties:
                    label: { type: string }
                    variant: { type: string, enum: [primary, secondary] }

/schemas/components/{componentType}:
  parameters:
    - name: componentType
      in: path
      required: true
      schema:
        type: string
  get:
    summary: Get schema for specific component type
    operationId: getComponentSchema
    responses:
      '200':
        description: JSON Schema for the component type
        content:
          application/json:
            schema:
              type: object
              description: JSON Schema
      '404':
        description: Unknown component type

/schemas/protocol:
  get:
    summary: Get protocol message schemas
    operationId: getProtocolSchemas
    description: |
      Returns JSON Schemas for protocol messages (actions, events, etc.).
      Useful for validation and code generation.
    responses:
      '200':
        description: Protocol message schemas
        content:
          application/json:
            schema:
              type: object
              properties:
                action:
                  type: object
                  description: Action message schema
                event:
                  type: object
                  description: Event message schema
                surface:
                  type: object
                  description: Surface schema
```

Include description explaining:
- Schema exchange enables validation on both sides
- Frontend defines what component types it supports
- Backend validates before sending, frontend validates on receive
- Enables code generation for both sides
  </action>
  <verify>
- File exists at spec/openapi/paths/schemas.yaml
- /schemas/components endpoint returns map of type to JSON Schema
- /schemas/protocol endpoint returns protocol message schemas
  </verify>
  <done>Schema discovery endpoint enables frontend-backend schema exchange</done>

Title: "The Backend-Drives-Frontend Pattern"

Structure:
1. **The Core Idea** (1 paragraph)
   - Backend returns configuration, frontend renders it
   - Any backend implementing the spec works with the frontend
   - No frontend code changes needed for new applications

2. **How It Works** (visual flow)
   ```
   Backend                          Frontend
   --------                         --------
   /navigation ----------------->   Renders menu/tabs
   /plugins/{id}/config -------->   Renders form fields or table columns
   /plugins/{id}/data ---------->   Fills in actual data
   /plugins/{id}/data (POST) <---   User submits, backend processes
   ```

3. **Plugin Types** (brief descriptions)
   - Form: Backend defines fields, validation rules, layout
   - Table: Backend defines columns, frontend handles scrolling/display
   - Action: Backend defines button, frontend triggers it
   - HTML: Backend returns rendered content
   - Cardlist: Backend defines card structure

4. **A Complete Example** (end-to-end)
   Show JSON for a simple "User Settings" form:
   - GET /navigation response showing "Settings" menu item
   - GET /plugins/user-settings/config response with form fields
   - GET /plugins/user-settings/data response with current values
   - POST /plugins/user-settings/data request to save changes

5. **Key Principles**
   - Frontend is generic, backend is specific
   - Configuration is lazy-loaded (only fetch what's needed)
   - Errors are standardized (RFC 9457)
   - Real-time via WebSocket triggers REST calls

**spec/docs/websocket.md:**

Title: "WebSocket Notification Protocol"

1. **Purpose**
   - WebSocket provides push notifications from backend to frontend
   - Notifications trigger REST calls - they don't carry data themselves
   - Pattern: "Something changed" -> Frontend re-fetches via REST

2. **Connection**
   - Endpoint: ws://host/api/v1/ws (or wss:// for TLS)
   - Authentication: Include JWT in connection URL or first message
   - Reconnection: Frontend handles with exponential backoff

3. **Message Format**
   Following JSON event-based convention from research:
   ```json
   ["eventName", {"key": "value"}]
   ```
   - First element: Event type string
   - Second element: Event payload object

4. **Event Types**

   `plugin_updated`:
   ```json
   ["plugin_updated", {
     "pluginId": "user-form-123",
     "action": "config_changed"
   }]
   ```
   Frontend should: Re-fetch /plugins/{pluginId}/config

   `plugin_data_changed`:
   ```json
   ["plugin_data_changed", {
     "pluginId": "user-table-456",
     "action": "row_added"
   }]
   ```
   Frontend should: Re-fetch /plugins/{pluginId}/data

   `session_expired`:
   ```json
   ["session_expired", {
     "reason": "timeout"
   }]
   ```
   Frontend should: Redirect to login

   `notification`:
   ```json
   ["notification", {
     "type": "info",
     "title": "Update Available",
     "message": "A new version is available"
   }]
   ```
   Frontend should: Display toast/banner

5. **Backend Implementation Notes**
   - Broadcast to all connections for a user/session
   - Don't include sensitive data in events (just triggers)
   - Keep events small and frequent rather than large and rare

**spec/docs/versioning.md:**

Title: "Protocol Versioning"

1. **Version Scheme**
   - URL path versioning: /api/v1/...
   - Current version: v1
   - Version appears in all endpoints

2. **What Constitutes a Breaking Change**
   - Removing or renaming an endpoint
   - Removing or renaming a required field
   - Changing field type
   - Changing error code semantics

3. **Non-Breaking Changes** (allowed without version bump)
   - Adding new optional fields
   - Adding new endpoints
   - Adding new error codes
   - Adding new plugin types
   - Plugin-specific config schema changes (validated internally)

4. **Compatibility Policy**
   - Backend and frontend deploy together (mono-repo or coordinated releases)
   - No deprecation window needed for v1
   - Future versions: N-1 compatibility (v2 backend supports v1 endpoints)

5. **OpenAPI Spec Hosting**
   - Spec served at: GET /api/v1/openapi.json
   - Always current for the running backend version
   - Clients can fetch and cache

# CallBackery Protocol

A capability-based protocol for server-driven user interfaces.

## Overview

The CallBackery Protocol defines how a backend configures and controls a frontend through standardized messages. The protocol is **capability-based**, not type-based: it defines surfaces, components, data binding, actions, and events without prescribing specific UI elements like forms or tables.

**Core principle:** The backend drives the frontend through a standardized, documented protocol. Any backend implementing the specification works with any compliant frontend.

## Documentation

- [Mental Model](./mental-model.md) - Understanding the puppet-master architecture
- [Quick Start](./quick-start.md) - Get oriented in 5 minutes

## Specification

- [OpenAPI Specification](../../spec/openapi/openapi.yaml) - Complete REST API definition
- [WebSocket Specification](../../spec/openapi/websocket.yaml) - Real-time event protocol

## Key Concepts

### Capability-Based Design

The protocol defines capabilities, not specific UI types:

| Capability | What it enables |
|------------|-----------------|
| **Surface** | Renderable container (main view, modal, toast) |
| **Component Tree** | UI structure (any component type) |
| **Data Model** | Application state with path-based binding |
| **Actions** | User intent (submit, click, navigate) |
| **Events** | Backend signals (data changed, notification) |

This design allows any frontend plugin (forms, tables, charts, custom widgets) to be built without protocol changes.

### The Protocol "Language"

Think of the protocol as a language:
- **Grammar** (message structure) is fixed and well-defined
- **Stories** (what you build with it) are unlimited

The protocol specifies how messages are structured, not what UI elements exist.

## Versioning

- **Protocol version:** URL path (`/api/v1/...`)
- **Schema versions:** Component schemas can evolve independently
- **Compatibility:** Backend and frontend deploy together (no deprecation window)

See [Versioning](./mental-model.md#versioning) for details.

## Implementation Status

- [ ] OpenAPI Specification - In Progress
- [ ] WebSocket Specification - In Progress
- [ ] Rust/Axum Reference Backend - Phase 2
- [ ] Svelte 5 Frontend - Phase 3

# Mental Model: Backend as Puppet Master

Understanding how the CallBackery Protocol works.

## The Puppet Theater Analogy

Imagine a puppet theater:

- **Puppet Master (Backend):** Controls what happens, pulls the strings
- **Smart Puppet (Frontend):** Performs actions skillfully, handles details
- **Script (Protocol):** The language they use to communicate
- **Strings (Messages):** How commands flow from master to puppet

The backend tells the frontend WHAT to show and WHAT to do. The frontend decides HOW to render it beautifully and HOW to handle user interactions smoothly.

### What the Puppet Master Controls

The backend controls the **content and behavior**:

- What surfaces exist (main view, modals, panels)
- What components appear in each surface
- What data is available
- How user actions are processed
- When to show notifications

### What the Smart Puppet Handles

The frontend handles the **presentation and interaction**:

- How to render each component type
- Animations, transitions, visual polish
- Accessibility (keyboard navigation, screen readers)
- Local input validation (format checking before submit)
- Responsive layout adaptation

## Capability-Based Protocol

### Why Not Define Specific UI Types?

Consider two approaches:

**Type-based (BAD):**
```
Protocol defines: TextField, NumberField, DateField, SelectField...
Problem: Need a CreditCardField? Change the protocol!
```

**Capability-based (GOOD):**
```
Protocol defines: Components have type (string), properties (object), children (IDs)
Solution: Frontend adds CreditCardField component, no protocol change!
```

The protocol provides building blocks. What you build is up to you.

### The Fundamental Atoms

Every server-driven UI needs these irreducible building blocks:

1. **Surface** - WHERE to show things
   - Main view, modal, sidebar, toast area
   - Each has its own component tree and data scope

2. **Component Tree** - WHAT to show
   - Flat list with ID references (adjacency list)
   - Any component type (open set, not enum)
   - Properties defined by frontend, not protocol

3. **Data Model** - VALUES to display/edit
   - JSON with path-based binding (JSON Pointer)
   - Components reference paths, not raw values
   - Changes propagate via binding

4. **Actions** - WHAT user wants to do
   - User intent (submit, click, navigate)
   - Backend decides response
   - Context resolved at send time

5. **Events** - WHAT backend wants to tell frontend
   - Triggers, not data carriers
   - Frontend re-fetches via REST
   - Real-time via WebSocket

6. **Validation** - IS THIS CORRECT?
   - Generic path-based errors
   - Not domain-specific codes
   - Keyword indicates validation type

## Communication Flow

### Configuration Flow

```
1. User navigates to a plugin
2. Frontend: GET /config/{pluginId}
3. Backend: Returns Surface with ComponentTree + DataModel
4. Frontend: Renders components, binds data
5. User sees the configured UI
```

### Action Flow

```
1. User interacts (clicks button, submits form)
2. Frontend: POST /actions with Action message
3. Backend: Processes action, decides response
4. Backend: Returns ActionResponse (update surface, show modal, navigate, etc.)
5. Frontend: Applies response effects
```

### Event Flow

```
1. Something changes on backend (data updated, session expires)
2. Backend: Sends Event via WebSocket
3. Frontend: Receives event
4. Frontend: Re-fetches data via REST (events are triggers)
5. Frontend: Updates UI with fresh data
```

### Re-configuration Flow

For dynamic UIs (field dependencies, conditional visibility):

```
1. User changes a field value
2. Frontend: POST /config/{pluginId}/reconfig with currentData
3. Backend: "Given this state, here's the new UI"
4. Frontend: Updates component tree (show/hide fields, etc.)
```

## Schema Exchange

### Why Schema Exchange?

The protocol doesn't define what "text-field" looks like. So how does the backend know what properties to send?

**Answer:** The frontend publishes schemas for component types it supports.

```
1. Frontend: Implements text-field, button, data-table components
2. Frontend: Publishes JSON Schemas at GET /schemas/components
3. Backend: Fetches schemas for components it wants to use
4. Backend: Validates config against schema before sending
5. Frontend: Validates received config against own schema
```

Both sides validate. Errors are caught before runtime.

### Schema Discovery Endpoint

```
GET /schemas/components
{
  "text-field": {
    "type": "object",
    "properties": {
      "label": { "type": "string" },
      "placeholder": { "type": "string" },
      "required": { "type": "boolean" }
    }
  },
  "button": {
    "type": "object",
    "properties": {
      "label": { "type": "string" },
      "variant": { "type": "string", "enum": ["primary", "secondary"] }
    }
  }
}
```

## Versioning

### Protocol Version

The protocol version is in the URL path:

```
/api/v1/config/users
/api/v2/config/users  (future)
```

Major version changes indicate breaking changes to message structure.

### Schema Versions

Component schemas can evolve independently:

```
GET /schemas/components?version=2
```

Frontend publishes which schema versions it supports. Backend uses compatible version.

### Compatibility Policy

For v1: **Backend and frontend deploy together.**

No deprecation window. When protocol changes, both sides update. This simplifies development and avoids version negotiation complexity.

Future consideration: If multiple frontends exist, introduce version negotiation.

## Anti-Patterns

### Don't: Embed UI Logic in Backend

**Bad:** Backend sends CSS classes, pixel dimensions, animation parameters
**Good:** Backend sends semantic properties (level: "warning"), frontend decides visual treatment

### Don't: Make Events Data Carriers

**Bad:** Event contains all updated data in payload
**Good:** Event is a trigger, frontend re-fetches via REST

### Don't: Enumerate Component Types in Protocol

**Bad:** Protocol defines TextField, DateField, NumberField as enum
**Good:** Protocol defines component type as string (open set)

### Don't: Define Domain-Specific Error Codes

**Bad:** Error codes like REQUIRED, TOO_SHORT, INVALID_EMAIL
**Good:** Generic errors with path, message, keyword (from JSON Schema)

# Quick Start: CallBackery Protocol

Get oriented in 5 minutes.

## The One-Sentence Summary

**The backend sends a component tree, the frontend renders it.**

That's the core idea. Everything else is details.

## The Core Loop

```
┌─────────────────────────────────────────────────────────┐
│                        Backend                          │
│  1. Receives request (config, action, data)             │
│  2. Decides what UI should look like                    │
│  3. Returns Surface with ComponentTree + Data           │
└────────────────────────┬────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────┐
│                        Frontend                         │
│  4. Receives Surface                                    │
│  5. Renders ComponentTree using registered components   │
│  6. Binds Data to components via JSON Pointer paths     │
│  7. User interacts → Action sent to backend → Repeat    │
└─────────────────────────────────────────────────────────┘
```

## What a Surface Looks Like

```json
{
  "id": "main",
  "type": "main",
  "componentTree": {
    "rootId": "form-1",
    "components": {
      "form-1": {
        "id": "form-1",
        "type": "container",
        "children": ["field-1", "field-2", "btn-1"]
      },
      "field-1": {
        "id": "field-1",
        "type": "text-field",
        "properties": { "label": "Name" },
        "dataPath": "/form/name"
      },
      "field-2": {
        "id": "field-2",
        "type": "text-field",
        "properties": { "label": "Email" },
        "dataPath": "/form/email"
      },
      "btn-1": {
        "id": "btn-1",
        "type": "button",
        "properties": { "label": "Submit", "action": "submit" }
      }
    }
  },
  "data": {
    "form": {
      "name": "",
      "email": ""
    }
  }
}
```

**Key points:**
- `componentTree` is a flat map (adjacency list), not nested
- `children` are ID references, not embedded objects
- `dataPath` is a JSON Pointer (RFC 6901)
- `type` is a string (frontend decides what "text-field" means)

## What an Action Looks Like

```json
{
  "action": {
    "type": "submit",
    "source": "btn-1",
    "surfaceId": "main",
    "context": {
      "form": {
        "name": "John Doe",
        "email": "john@example.com"
      }
    },
    "timestamp": "2026-01-23T12:00:00Z"
  }
}
```

**Key points:**
- `type` identifies what happened (submit, click, navigate)
- `source` is the component that triggered it
- `context` contains resolved data (not paths)

## What an Event Looks Like

```json
{
  "type": "data_changed",
  "surfaceId": "main",
  "payload": {
    "pluginId": "users-table"
  },
  "timestamp": "2026-01-23T12:00:00Z"
}
```

**Key points:**
- Events are triggers (small payloads)
- Frontend re-fetches data via REST
- Delivered via WebSocket

## Minimum Viable Implementation

### Backend Must:

1. Serve `GET /config` - Return app navigation and main surface
2. Serve `GET /config/{pluginId}` - Return plugin surface
3. Handle `POST /actions` - Process user actions
4. Optionally: WebSocket for real-time events

### Frontend Must:

1. Fetch config on load
2. Render component tree (map type to component)
3. Bind data via JSON Pointer paths
4. Send actions on user interaction
5. Optionally: Connect WebSocket for events

## Checklist: Is Your Implementation Correct?

- [ ] Component tree is flat (adjacency list), not nested JSON
- [ ] Component type is string (not enum)
- [ ] Data binding uses JSON Pointer paths
- [ ] Actions include source component ID
- [ ] Events are triggers (frontend re-fetches data)
- [ ] Validation errors use path + message + keyword

## Next Steps

- Read [Mental Model](./mental-model.md) for deeper understanding
- Review [OpenAPI Spec](../../spec/openapi/openapi.yaml) for complete API
- Review [WebSocket Spec](../../spec/openapi/websocket.yaml) for events

      ## Versioning Scheme

      ### Protocol Version
      The protocol version is embedded in the URL path:
      - `/api/v1/...` - Current stable version
      - `/api/v2/...` - Future major version (breaking changes)

      Version changes when:
      - Message envelope structure changes
      - Required fields are added/removed
      - Endpoint paths change

      ### Schema Versions
      Component schemas (published by frontend) version independently:
      - `GET /schemas/components?version=N`
      - Frontend publishes which versions it supports
      - Backend uses compatible version

      ### Compatibility Policy
      For v1: Backend and frontend deploy together.
      - No deprecation window
      - No version negotiation
      - When protocol changes, both sides update simultaneously

      This simplifies development and testing. Version negotiation may
      be introduced in future if multiple frontends need to coexist.

      ### Breaking vs Non-Breaking Changes

      **Breaking (requires major version):**
      - Adding required fields to existing messages
      - Removing fields
      - Changing field types
      - Renaming endpoints

      **Non-Breaking (minor/patch):**
      - Adding optional fields
      - Adding new endpoints
      - Adding new event types
      - Documentation updates
    policy: deploy-together
    current: v1
```

Title: "Error Code Reference"

1. **Error Response Format**
   Reference RFC 9457 Problem Details structure:
   ```json
   {
     "type": "https://api.callbackery.com/problems/auth-invalid-credentials",
     "title": "Authentication Failed",
     "status": 401,
     "detail": "The email or password provided is incorrect",
     "instance": "/api/v1/auth/login"
   }
   ```

2. **Error Categories**
   Organize by HTTP status code and domain:

   **Authentication Errors (401)**
   | Code | Type URI | Title | When |
   |------|----------|-------|------|
   | AUTH_INVALID_CREDENTIALS | .../auth-invalid-credentials | Authentication Failed | Login with wrong email/password |
   | AUTH_TOKEN_EXPIRED | .../auth-token-expired | Token Expired | JWT has expired |
   | AUTH_TOKEN_INVALID | .../auth-token-invalid | Invalid Token | JWT malformed or signature invalid |
   | AUTH_REQUIRED | .../auth-required | Authentication Required | No token provided for protected endpoint |

   **Authorization Errors (403)**
   | Code | Type URI | Title | When |
   |------|----------|-------|------|
   | AUTHZ_FORBIDDEN | .../authz-forbidden | Access Denied | User lacks permission for resource |
   | AUTHZ_PLUGIN_ACCESS | .../authz-plugin-access | Plugin Access Denied | User cannot access this plugin |

   **Not Found Errors (404)**
   | Code | Type URI | Title | When |
   |------|----------|-------|------|
   | PLUGIN_NOT_FOUND | .../plugin-not-found | Plugin Not Found | Plugin ID doesn't exist |
   | RESOURCE_NOT_FOUND | .../resource-not-found | Resource Not Found | Generic resource not found |
   | NAVIGATION_NOT_FOUND | .../navigation-not-found | Navigation Item Not Found | Nav item ID doesn't exist |

   **Validation Errors (422)**
   | Code | Type URI | Title | When |
   |------|----------|-------|------|
   | VALIDATION_FAILED | .../validation-failed | Validation Failed | One or more fields invalid |

   Field-level codes (in errors array):
   | Code | Meaning | Example |
   |------|---------|---------|
   | REQUIRED | Field is required but missing | "Email is required" |
   | INVALID_FORMAT | Field format is wrong | "Invalid email format" |
   | OUT_OF_RANGE | Value outside allowed range | "Age must be 0-150" |
   | INVALID_OPTION | Value not in allowed set | "Status must be active or inactive" |
   | TOO_SHORT | String too short | "Password must be at least 8 characters" |
   | TOO_LONG | String too long | "Name cannot exceed 100 characters" |
   | INVALID_TYPE | Wrong data type | "Expected number, got string" |
   | UNIQUE_VIOLATION | Value already exists | "Email already registered" |

   **Server Errors (500)**
   | Code | Type URI | Title | When |
   |------|----------|-------|------|
   | INTERNAL_ERROR | .../internal-error | Internal Server Error | Unexpected server failure |
   | DATABASE_ERROR | .../database-error | Database Error | Database operation failed |

3. **Using Errors in Responses**

   Single error example:
   ```json
   {
     "type": "https://api.callbackery.com/problems/plugin-not-found",
     "title": "Plugin Not Found",
     "status": 404,
     "detail": "Plugin with ID 'user-form-xyz' does not exist",
     "instance": "/api/v1/plugins/user-form-xyz/config"
   }
   ```

   Validation error with field details:
   ```json
   {
     "type": "https://api.callbackery.com/problems/validation-failed",
     "title": "Validation Failed",
     "status": 422,
     "detail": "The request contains invalid fields",
     "instance": "/api/v1/plugins/user-form/data",
     "errors": [
       {"field": "email", "code": "REQUIRED", "message": "Email is required"},
       {"field": "age", "code": "OUT_OF_RANGE", "message": "Age must be between 0 and 150"}
     ]
   }
   ```

4. **Error Handling Guidelines**
   - Always return appropriate HTTP status code
   - Include request_id in server errors for debugging
   - Field messages should be user-friendly fallbacks (frontend can translate codes)
   - Return ALL validation errors together (don't fail on first)

5. **Extending Error Codes**
   - New error codes can be added without version bump
   - Custom error types must use same structure
   - Type URIs should be stable (don't change meaning)

Create GitHub Actions workflow for OpenAPI validation:

**.github/workflows/validate-spec.yml:**

```yaml
name: Validate OpenAPI Specification

on:
  push:
    paths:
      - 'spec/**'
      - '.spectral.yaml'
  pull_request:
    paths:
      - 'spec/**'
      - '.spectral.yaml'

jobs:
  validate:
    name: Lint and Validate
    runs-on: ubuntu-latest

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Install Spectral
        run: npm install -g @stoplight/spectral-cli

      - name: Lint OpenAPI Specification
        run: spectral lint spec/openapi.yaml --ruleset .spectral.yaml --format stylish

      - name: Validate OpenAPI structure
        run: |
          # Check that spec is valid OpenAPI 3.1
          npx @redocly/cli lint spec/openapi.yaml --skip-rule=no-unused-components

      - name: Check for breaking changes (on PR only)
        if: github.event_name == 'pull_request'
        run: |
          # Install oasdiff for breaking change detection
          npm install -g oasdiff
          # Fetch base branch spec
          git fetch origin ${{ github.base_ref }}
          git show origin/${{ github.base_ref }}:spec/openapi.yaml > /tmp/base-spec.yaml 2>/dev/null || echo "No base spec found, skipping diff"
          # Compare if base exists
          if [ -f /tmp/base-spec.yaml ]; then
            oasdiff breaking /tmp/base-spec.yaml spec/openapi.yaml || echo "Breaking changes detected - review required"
          fi
```

Workflow features:
- Triggers on changes to spec/ or .spectral.yaml
- Runs Spectral linting with project ruleset
- Validates OpenAPI structure with Redocly CLI
- Detects breaking changes on PRs (informational)
- Uses Node 20 for latest tool compatibility

Create .github/workflows directory if it doesn't exist.

<task type="auto">
  <name>Task 2: Define component model schema (ID-based adjacency list)</name>
  <files>spec/openapi/schemas/component.yaml</files>
  <action>
Create component model schema following A2UI adjacency list pattern.

Key design from research:
- Components as FLAT LIST with ID references (not nested JSON)
- Each component has: id, type, properties, children (ID references)
- type is STRING (open set - allows any component type without protocol change)
- properties is OBJECT (schema defined by frontend, not protocol)
- children is ARRAY of STRING (component IDs, not nested objects)

Schema definitions to include:

1. **ComponentId** - String type for component identifiers

2. **ComponentNode** - Single component in the tree:
   ```yaml
   type: object
   required: [id, type]
   properties:
     id:
       $ref: '#/components/schemas/ComponentId'
       description: Unique identifier within the surface
     type:
       type: string
       description: Component type (open set - frontend determines valid types)
       examples: ["text-field", "button", "data-table", "container"]
     properties:
       type: object
       additionalProperties: true
       description: Component-specific properties (schema defined by frontend)
     children:
       type: array
       items:
         $ref: '#/components/schemas/ComponentId'
       description: IDs of child components (adjacency list pattern)
     dataPath:
       type: string
       description: JSON Pointer to data model value (RFC 6901)
       pattern: "^(/[^/~]*(~[01][^/~]*)*)*$"
       examples: ["/user/email", "/form/fields/0/value"]
   ```

3. **ComponentTree** - Collection of components for a surface:
   ```yaml
   type: object
   required: [rootId, components]
   properties:
     rootId:
       $ref: '#/components/schemas/ComponentId'
       description: ID of the root component
     components:
       type: object
       additionalProperties:
         $ref: '#/components/schemas/ComponentNode'
       description: Map of component ID to component definition
   ```

Include detailed descriptions explaining:
- Why adjacency list (LLM-friendly, easy patching, streaming-compatible)
- Why type is string not enum (open set enables future components)
- How children references work
  </action>
  <verify>
- File exists at spec/openapi/schemas/component.yaml
- YAML is valid
- Contains ComponentNode, ComponentTree, ComponentId schemas
- ComponentNode.type is type: string (not enum)
- ComponentNode.children is array of ComponentId (not nested objects)
  </verify>
  <done>Component model uses flat ID-based adjacency list, type is open string, children are ID references</done>
</task>

<task type="auto">
  <name>Task 3: Define data model and binding schemas</name>
  <files>spec/openapi/schemas/data.yaml</files>
  <action>
Create data model schema with JSON Pointer binding support.

Key design from research:
- Data model is JSON with path-based binding
- Paths use JSON Pointer (RFC 6901): /user/email, /items/0/name
- Two-way binding: frontend updates data, backend notified
- Scoped data for collections (list items have own context)

Schema definitions to include:

1. **JsonPointer** - String type with JSON Pointer format:
   ```yaml
   type: string
   description: JSON Pointer (RFC 6901) path to data value
   pattern: "^(/[^/~]*(~[01][^/~]*)*)*$"
   examples:
     - "/user/email"
     - "/form/fields/0/value"
     - "/items/~1special~0char"
   ```

2. **DataModel** - The actual data container:
   ```yaml
   type: object
   additionalProperties: true
   description: |
     Application data that components display/edit.
     Components reference data via JSON Pointer paths.
     Supports any JSON structure - schema not constrained by protocol.
   ```

3. **DataUpdate** - Message for changing data:
   ```yaml
   type: object
   required: [path, value]
   properties:
     path:
       $ref: '#/components/schemas/JsonPointer'
       description: Path to the value being updated
     value:
       description: New value (any JSON type)
     timestamp:
       type: string
       format: date-time
       description: When the update occurred (for conflict resolution)
   ```

4. **DataPatch** - Batch of updates:
   ```yaml
   type: object
   required: [updates]
   properties:
     updates:
       type: array
       items:
         $ref: '#/components/schemas/DataUpdate'
   ```

Include description explaining:
- JSON Pointer RFC 6901 reference
- Two-way binding semantics
- How components reference data via dataPath property
  </action>
  <verify>
- File exists at spec/openapi/schemas/data.yaml
- Contains JsonPointer, DataModel, DataUpdate, DataPatch schemas
- JsonPointer has RFC 6901 pattern
  </verify>
  <done>Data model supports JSON Pointer path-based binding with update semantics</done>
</task>

<task type="auto">
  <name>Task 4: Define message envelope and error schemas</name>
  <files>
    spec/openapi/schemas/envelope.yaml
    spec/openapi/schemas/error.yaml
  </files>
  <action>
Create message envelope and error response schemas.

**Message Envelope** (envelope.yaml):

Protocol messages have consistent wrapper structure.

1. **MessageEnvelope** - Generic message wrapper:
   ```yaml
   type: object
   required: [type, timestamp]
   properties:
     type:
       type: string
       description: Message type identifier
     timestamp:
       type: string
       format: date-time
       description: When message was created
     correlationId:
       type: string
       format: uuid
       description: Links request to response
     payload:
       description: Message-specific content (any JSON)
   ```

2. **ApiResponse** - Standard API response wrapper:
   ```yaml
   type: object
   required: [success]
   properties:
     success:
       type: boolean
     data:
       description: Response payload on success
     error:
       $ref: 'error.yaml#/components/schemas/ApiError'
   ```

**Error Schemas** (error.yaml):

Generic path-based errors (NOT domain-specific codes).

1. **ValidationError** - Single validation issue:
   ```yaml
   type: object
   required: [path, message]
   properties:
     path:
       $ref: 'data.yaml#/components/schemas/JsonPointer'
       description: Path to the invalid value
     message:
       type: string
       description: Human-readable error message
     keyword:
       type: string
       description: JSON Schema validation keyword that failed
       examples: ["required", "format", "minimum", "maxLength", "pattern"]
   ```

2. **ValidationErrors** - Collection of validation issues:
   ```yaml
   type: object
   required: [errors]
   properties:
     errors:
       type: array
       items:
         $ref: '#/components/schemas/ValidationError'
   ```

3. **ApiError** - General API error:
   ```yaml
   type: object
   required: [code, message]
   properties:
     code:
       type: string
       description: Error code for programmatic handling
       examples: ["VALIDATION_ERROR", "NOT_FOUND", "UNAUTHORIZED", "SERVER_ERROR"]
     message:
       type: string
       description: Human-readable error description
     details:
       $ref: '#/components/schemas/ValidationErrors'
       description: Detailed validation errors if applicable
   ```

Key design: Error codes are generic HTTP-level (VALIDATION_ERROR, NOT_FOUND), NOT domain-specific (REQUIRED, TOO_SHORT). Domain validation uses path + message + keyword pattern.
